# UDS 诊断协议入门

> 基于 ISO 14229-1 标准，结合本项目的 STM32 端实现讲解。读完你会理解 UDS 是什么、诊断仪怎么读取 ECU 数据、以及"诊断会话"的作用。

---

## 一、UDS 是干什么的

### 1.1 一个实际场景

你的车仪表盘亮了"检查发动机"灯。修车师傅接上一个诊断仪，屏幕上立刻显示：

```
故障码 P0301 — 1缸失火
冻结帧: 转速=3500rpm, 车速=80km/h, 冷却液=92°C
```

诊断仪（Tester）通过 CAN 总线向发动机 ECU 发了几个**UDS 请求**，ECU 回复了对应的数据。整个过程不需要拆车、不需要接线——插上 OBD-II 接口就行。

**UDS（Unified Diagnostic Services）就是诊断仪和 ECU 之间的"对话协议"。**

### 1.2 一句话总结

> UDS = 汽车诊断的标准语言。诊断仪问什么，ECU 答什么。不管是博世的 ECU 还是电装的 ECU，都说同一种 UDS 语言。

## 二、UDS 的工作原理

### 2.1 物理寻址

诊断仪通过两个 CAN ID 与 ECU 一对一通信。本项目两个节点各用不同的 ID 对：

```
节点 A (动力域):  请求 0x7E0 / 响应 0x7E8
节点 B (车身域):  请求 0x7E1 / 响应 0x7E9
```

这样诊断仪可以分别跟两个节点对话，不会冲突。

### 2.2 帧格式

```
请求帧 (CAN ID=0x7E0):
  byte0 = 服务ID (SID)
  byte1 = 子功能/参数高字节
  byte2 = 参数低字节
  ...

正响应 (CAN ID=0x7E8):
  byte0 = 服务ID + 0x40  (例: 0x22 → 0x62)
  byte1+ = 响应数据

负响应 (CAN ID=0x7E8):
  byte0 = 0x7F
  byte1 = 原始服务ID
  byte2 = 否定响应码 (NRC)  — 告诉你为什么失败了
```

**正响应规则**：SID + 0x40。如果请求是 0x22（读数据），正响应就是 0x62。

**负响应规则**：0x7F + 原始SID + 错误码。比如"我不支持这个服务" → `[0x7F, 0x22, 0x11]`。

## 三、本项目实现的四个服务

### 3.1 0x10 — 诊断会话控制

**作用**：切换 ECU 的工作模式。不同的会话提供不同级别的功能权限。

```
会话 0x01 — 默认会话: 基本功能（上电后的默认状态）
会话 0x02 — 编程会话: 允许刷写固件（本项目暂未用到）
会话 0x03 — 扩展会话: 允许读取更详细的诊断数据
```

**S3 超时**：切换到非默认会话后，如果没有收到任何诊断请求，**5 秒后自动切回默认会话**。这是安全机制——防止修车师傅诊断完忘了退出，ECU 一直留在高权限模式。

```
请求: [0x10, 0x03]           — 进入扩展会话
响应: [0x50, 0x03, 0, 50, 0] — 确认 + 时序参数
```

### 3.2 0x22 — 按 ID 读数据

**作用**：读取 ECU 的指定数据项（DID — Data Identifier）。每个 DID 是一个 16-bit 编号，对应一个具体数据。

本项目支持的 DID：

| DID | 节点 A 数据 | 节点 B 数据 | 用途 |
|---|---|---|---|---|
| 0xF1A0 | "1.0" | "1.0" | 软件版本号 |
| 0xF190 | "NODEA" | "NODEB" | VIN 码（模拟） |
| 0xF100 | "ST-A" | "ST-B" | ECU 序列号 |
| 0xF1B0 | 总线负载 | 总线负载 | CAN 总线健康 |
| 0xF1B1 | TEC 值 | TEC 值 | 发送错误计数 |
| 0xF1B2 | REC 值 | REC 值 | 接收错误计数 |
| 0xF1B3 | 累计帧数 | 累计帧数 | 运行统计 |

响应按 ISO 14229-1 标准格式：`[SID+0x40] [DID高] [DID低] [数据...]`。

**示例**：

```
请求: [0x22, 0xF1, 0x90]     — 读 VIN 码
响应: [0x62, 0xF1, 0x90, 'N','O','D','E','A']   ← 8字节
```

如果请求不支持的 DID（比如 0xF200），回复 NRC 0x31（请求超出范围）。

### 3.3 0x3E — 诊断会话保持

**作用**：告诉 ECU "我还活着，别切回默认会话"。

诊断仪通常每秒发一次 0x3E，刷新 S3 超时定时器。如果不发，5 秒后 ECU 自动切回默认会话。

```
请求: [0x3E, 0x00]           — 子功能 0x00，不要求回复
请求: [0x3E, 0x01]           — 子功能非零，回复 ACK
响应: [0x7E, 0x01]            — ACK
```

### 3.4 0x11 — ECU 复位

**作用**：远程让 ECU 重启。支持两种复位类型：

```
0x01 — 硬复位: 等效于断电重启
0x03 — 软复位: 软件复位，不断电
```

```
请求: [0x11, 0x01]   — 请求硬复位
响应: [0x51, 0x01]   — 确认，100ms 后执行 NVIC_SystemReset()
```

## 四、本项目中的完整交互示例

诊断仪和节点 A（0x7E0/0x7E8）的典型对话：

```
1. Tester → NodeA:  [0x10, 0x03]            "进入扩展会话"
   NodeA → Tester:  [0x50, 0x03, 0, 50, 200] "好的"

2. Tester → NodeA:  [0x22, 0xF1, 0x90]      "读 VIN"
   NodeA → Tester:  [0x62, 0xF1, 0x90, 'N','O','D','E','A']  "NODEA"

3. Tester → NodeA:  [0x22, 0xF1, 0xB1]      "读 TEC"
   NodeA → Tester:  [0x62, 0xF1, 0xB1, 0, 0]  "TEC=0"

4. Tester → NodeA:  [0x3E, 0x00]             "会话保持"(不回复)

5. Tester → NodeA:  [0x11, 0x01]             "硬复位"
   NodeA → Tester:  [0x51, 0x01]             "收到，20ms后重启"
```

节点 B 用相同命令格式，CAN ID 换成 `7E1`/`7E9`。

## 五、UDS 和周期诊断帧的区别

本项目现在有**两种**诊断机制：

| 特性 | 周期诊断帧 (0x300) | UDS (0x7E0/0x7E8) |
|---|---|---|
| 触发方式 | 自动 10s 周期发送 | 请求-响应（主控主动问） |
| 数据内容 | TEC/REC/总线负载/发送计数 | 同上 + 软件版本/VIN/序列号 |
| 会话控制 | 无 | 支持默认/编程/扩展会话 |
| 远程复位 | 不支持 | 支持 |
| 应用场景 | 持续监控 | 修车/调试/刷固件 |
| 标准化程度 | 项目自定义 | ISO 14229-1 国际标准 |

两者互补——0x300 用于持续性健康监控，UDS 用于按需诊断和远程控制。

## 六、实操指南

### 6.1 你需要什么

- STM32 已经烧录并连上 CAN 总线
- 一台 Linux 电脑（或虚拟机），安装 `can-utils`：

```bash
sudo apt install can-utils
```

### 6.2 创建虚拟 CAN 接口

如果你用 x86 电脑开发（没有真实 CAN 硬件），创建一条虚拟 CAN 总线：

```bash
sudo ip link add dev vcan0 type vcan
sudo ip link set up vcan0
```

`vcan0` 就是你的"CAN 总线"，可以像真实 CAN 一样收发数据。

### 6.3 先确认 STM32 在说话

```bash
candump vcan0
```

如果 STM32 正常工作，屏幕上会不断刷出类似这样的东西：

```
vcan0  180  [3]  31 96 00     ← 超声波距离
vcan0  181  [2]  11 02        ← 档位
vcan0  200  [3]  21 FF 00     ← 温度
```

这说明 CAN 通信正常。

### 6.4 向 STM32 发第一条 UDS 命令

开两个终端窗口。

**窗口 1 — 监听回复**：

```bash
candump vcan0,7E8:FFFFFFFF
```

这行命令的意思是："只看 CAN ID 为 `7E8` 的帧（UDS 回复），其他全忽略"。

**窗口 2 — 发送命令**：

```bash
cansend vcan0 7E0#22.F1.90
```

这行命令是什么意思？

```
cansend vcan0         → 往 vcan0 这条 CAN 总线发数据
7E0                   → CAN ID，0x7E0 是"诊断请求"
#                     → 分隔符，后面是数据
22                    → 第1个字节：22 就是"读数据"这个服务
F1.90                 → 第2-3个字节：F190 是"车辆识别码(VIN)"这个数据项
```

发完之后，窗口 1 会显示类似：

```
vcan0  7E8  [13]  62 F1 90 53 54 4D 33 32 2D 4E 4F 44 45 41
```

解读：
```
7E8         → 这是回复（ECU→诊断仪）
[13]        → 回复了 13 个字节
62          → 0x22 + 0x40 = 正响应
F1 90       → 你问的那个数据项
53 54 ...   → 剩余是数据：S-T-M-3-2---N-O-D-E-A
```

### 6.5 试试其他命令

```bash
# 节点 A（0x7E0 请求）
cansend can0 7E0#22.F1.00    # 读序列号 → "ST-A"
cansend can0 7E0#22.F1.90    # 读 VIN → "NODEA"
cansend can0 7E0#22.F1.B1    # 读 TEC
cansend can0 7E0#11.01       # 复位

# 节点 B（0x7E1 请求）
cansend can0 7E1#22.F1.00    # 读序列号 → "ST-B"
cansend can0 7E1#22.F1.90    # 读 VIN → "NODEB"
cansend can0 7E1#11.01       # 复位

# 监听（两个窗口分别开）
candump can0,7E8:FFFFFFFF    # 看 NodeA 回复
candump can0,7E9:FFFFFFFF    # 看 NodeB 回复
```

### 6.6 从车载终端（Qt）发 UDS

如果你的 vehicle-terminal 在运行，在"CAN 调试"页面：

1. CAN ID 填 `7E0`
2. 数据填 `22 F1 90`
3. 点发送
4. 接收区会看到 `7E8` 的回复

## 七、面试话术

> "UDS 是 ISO 14229-1 定义的汽车诊断标准协议。我在 STM32 端实现了四个基础 UDS 服务——会话控制(0x10)、按 ID 读数据(0x22)、会话保持(0x3E)和 ECU 复位(0x11)。诊断仪通过 0x7E0 发请求，ECU 在 0x7E8 回复，正响应 SID+0x40，负响应 0x7F+NRC。实现了 S3 超时自动切回默认会话的安全机制。硬件层面新增了一个 CAN 滤波器组精确匹配 0x7E0，与现有控制帧滤波器并行工作。"
