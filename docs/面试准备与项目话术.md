## 二、项目介绍（1分钟版本）

> 我做的是一个基于 CAN 总线的分布式智能座舱原型系统。下位机两块 STM32F103 分别模拟动力域和车身域 ECU，跑 FreeRTOS 管理 7 个任务，通过 250kbps CAN 总线周期上报 8 类传感器+执行器反馈数据并接收主控指令。上位机是 Qt 5.15 车载终端，跑在 ARM Linux 上，集成仪表盘、百度地图导航、倒车影像和 CAN 调试工具。
>
> 技术亮点有三个：一是参考 AUTOSAR 架构设计了 4 层通信栈（SWC→RTE→COM→Driver），信号打包/解包通过描述符表驱动而非硬编码字节偏移；二是实现了 E2E Profile 1 保护——每个传感器帧带 valid flag + 4-bit 滚动计数器，Qt 端做丢帧检测，3 秒心跳超时检测节点离线；三是建立了代码生成管线——JSON 定义协议，Python 脚本同时产出 C 头文件和标准 DBC 文件，与 SavvyCAN/cantools 等工业工具链兼容。

## 四、高频问题 + 回答模板

### Q1: 介绍一下你的项目

**回答策略**：先一句话概括，再展开技术深度。

> 我做的是一个分布式智能座舱系统，重点在 CAN 总线通信。下位机用两块 STM32 模拟动力域和车身域 ECU，上位机用 Qt 做车载终端。项目完整覆盖了从 CAN 控制器寄存器配置到应用层协议设计的技术栈。

> 技术深度主要在三块：一是参考 AUTOSAR 做了 4 层通信栈分层，信号打包/解包通过描述符表驱动，不硬编码字节偏移；二是实现了 E2E 保护——每个传感器帧带 valid flag 和独立滚动计数器，Qt 端做丢帧检测和心跳超时；三是用 Python 写了代码生成工具——JSON 定义协议 → 自动产出 C 头文件 + DBC 文件。

---

### Q2: CAN 总线 ID 分配为什么 bit[7] 用来区分控制/上报？

**回答策略**：先讲仲裁机制，再讲设计动机。

> CAN 总线仲裁是逐位"线与"机制——谁先发出显性位(0)谁就赢。所以 ID 越小优先级越高。我的 ID 分配按范围划分：控制帧 0x080-0x17F（最高优先级），上报帧 0x180-0x27F（次优先），诊断帧 0x300-0x37F（最低）。核心原则是控制帧的 ID 范围整体低于上报帧——控制指令在仲裁中永远先于传感器数据发出。

> 近似按 bit-field 理解：bit[10:8] 编码节点（001=A, 010=B, 011=诊断），bit[7] 大致区分方向（控制帧落在低半区）。但实际分配是按范围而非严格按位——比如 0x080（控制）bit[7]=1 但仍在控制区。面试时画范围图比纯位域更准确。

---

### Q3: 你项目中 4-bit 滚动计数器是干什么的？为什么 4-bit 而不是更大？

**回答策略**：先讲要解决什么问题，再讲为什么选 4-bit。

> 滚动计数器做的是丢帧检测——每次发帧时 counter+1，接收方检查是否连续递增。如果从 3 跳到 5，说明第 4 帧丢了。

> 为什么只用 4-bit？因为我把 counter 和 valid flag 合在一个字节里——bit0 是 valid，bit[7:4] 是 counter，零额外带宽开销。4-bit 循环周期是 16 帧，在 10Hz 发送频率下窗口 1.6 秒，足够检测连续丢帧。如果要检测更大的间隙，counter 循环绕回会让期望值计算混乱——但 16 帧都丢了，心跳早就超时了。所以 counter 检测短期丢失，心跳检测节点整体掉线，两者互补。

---

### Q4: 你怎么检测 STM32 节点死了？

**回答策略**：讲心跳机制的演进——从独立心跳帧到融合方案。

> 最早的设计用了独立心跳帧（最高优先级 ID），1Hz 周期只发一个"我还活着"。但心跳帧占用了宝贵的仲裁优先级，且心跳帧丢了就误报。

> 后来改成"任一上报帧到达就是心跳"——节点每 100ms 发 4 帧数据，任何一帧到达就 reset 计时器。Qt 端 1Hz 检查是否超过 3 秒没收到任何帧。好处是：不依赖单一帧类型（4 种帧独立发送），不占用额外 CAN ID 和总线带宽，心跳频率等效 10Hz 远优于 1Hz。

---

### Q5: CAN 发送时 while(1) 重试有什么问题？

**回答策略**：这是经典 RTOS 优先级翻转问题。

> CAN 任务是系统最高优先级(prio=4)。CAN 控制器进 BusOff 后邮箱永久不可用，HAL_CAN_AddTxMessage 持续返回 HAL_ERROR。如果写 while(1) 忙等 retry 不 taskYIELD——这个任务永远不会让出 CPU，所有 prio≤4 的任务全部饿死，系统表现为"死机"。

> 我设计了超时重试机制：最多重试 100 次(约 5ms)，每 10 次调用 taskYIELD()。5ms 后仍失败返回 -1，CAN 任务在下一个 20ms 周期自然重试。这样就保证其他任务不会被饿死。

---

### Q6: 你的 COM 层和 AUTOSAR COM 是什么关系？

**回答策略**：不用说"实现了 AUTOSAR"，说"参考了设计思想"。

> 不是实现了 AUTOSAR——AUTOSAR 全套需要商业工具链(Vector DaVinci/EB tresos)生成代码，授权费数十万。我是参考了它的分层设计思想：

> 描述符表驱动——`g_signal_desc[]` 数组，发送时通过 SignalId 查表获取 CAN ID 和字节偏移，接收时同样的表做逆向解包。这等价于 AUTOSAR 的 Signal Descriptor Table + Com_SendSignal/Com_ReceiveSignal 接口。

> 项目有 4 层通信栈：SWC(HomePage) → RTE(Rte) → COM(ComLayer/com_layer) → Driver(canmanager/can_drv)，对应 AUTOSAR 的 SWC → RTE → COM → CanIf → CanDrv 分层。

---

### Q7: 你遇到的最难的 bug 是什么？

**回答策略**：挑有深度的——ISR 栈溢出或 CAN 死锁。

> 最复杂的是 CAN 死锁。现象是系统运行一段时间后 OLED 停止刷新、按键无响应——看起来像硬件死机。Debug 发现 CAN 任务卡在发送循环里不 yield，BusOff 后邮箱不可用导致无限等待。根因是 FreeRTOS 优先级管理的边界 case——不是低优先级阻塞高优先级，而是最高优先级自己卡死自己，谁也救不了。

> 另外一个是 DHT11 校验和误判。C 语言的整数提升规则把 `uint8_t` 提升为 `int` 做加法，结果超出了 `uint8_t` 范围但隐式转换后正确。加了 `(uint8_t)` 截断才恢复正确的校验逻辑。

---

### Q8: 你在项目里怎么保证代码质量？

> 一是协议定义用 JSON 做单一数据源——改协议只要改一个文件，Python 脚本自动生成 C 头文件和 DBC，杜绝 C++/C/DBC 三份拷贝不同步的问题。

> 二是 Qt 端用了 15 个 `static_assert` 做编译期 CAN ID 校验——如果生成的头文件里 ID 值和预期不一致，编译直接报错。

> 三是传感器数据有效性做显式标记——读失败时设置 valid=0，接收端收到 invalid 不更新 UI，防止僵尸数据。

## 六、项目不足与改进方向

"如果继续做，你会怎么改进？"——面试官几乎必问的问题。

### 6.1 E2E CRC

> 目前只做了 counter + valid，没做应用层 CRC。可以加 CRC-8 覆盖 payload + counter，实现 AUTOSAR E2E Profile 1 完整校验——防止 COM 层打包/解包时字节错位。

### 6.2 双 CAN 总线 + 网关

> 汽车上动力域和车身域是两条独立 CAN 总线，速率不同(500kbps vs 125kbps)，中间有网关路由。目前单总线仿真没体现网关的 ID 转换和速率适配。

### 6.3 CAN FD 升级

> 标准 CAN 只能发 8 字节，CAN FD 可以发 64 字节。诊断帧(DLC=8)已经顶满，再加诊断数据(如 FreeRTOS 栈高水位)需要发两帧。

### 6.4 诊断 DTC 体系

> 目前诊断只上报 TEC/REC 计数器。完整的诊断需要 ISO 15765 (UDS) 协议栈——包括故障码(DTC)、冻结帧(Freeze Frame)、诊断会话(默认/扩展/编程)。

### 6.5 时间触发

> 当前纯周期+事件触发。如果多个节点需要严格同步(如同时采样 ADC)，需要 CAN TTCAN 时间触发机制。

## 七、项目总结（简历用）

**项目名称**：基于 CAN 总线的分布式智能座舱系统

**时间**：2025-2026

**技术栈**：C (C99) / C++ (Qt 5.15) / Python / FreeRTOS / STM32F103 / ARM Linux / SocketCAN / CAN 250kbps / DBC

**工作内容**：
- 设计 CAN 协议 ID 分配方案（bit[10:8]=node, bit[7]=dir），控制帧仲裁优先级高于上报帧
- 参考 AUTOSAR 架构实现 4 层通信栈（SWC→RTE→COM→Driver），信号通过描述符表驱动打包/解包
- 实现 E2E 数据保护：valid flag + 4-bit 滚动计数器 + 3s 心跳超时节点存活检测
- 修复 CAN 发送死锁、ISR 栈溢出、BusOff 检测等 15+ 处缺陷
- 搭建代码生成管线：JSON 协议定义 → Python 生成 C 头文件 + DBC 文件
- Qt 端设计 Meyer's 单例 + RTE 模式切换(CAN/Mock)，x86 无硬件可完成所有 UI 测试
