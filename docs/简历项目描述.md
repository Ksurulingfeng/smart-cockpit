# 简历项目描述

## 基于 CAN 总线的分布式智能座舱系统

**技术栈**：IMX6ULL (Linux/Qt) + STM32 (FreeRTOS) × 2 + CAN 250kbps

独立设计并实现 STM32+FreeRTOS 双节点实时固件与 Qt/Linux 上位机，覆盖嵌入式、通信协议、桌面应用三层全栈。参考 AUTOSAR Classic Platform 架构自研分层通信中间件，支撑 15 条 CAN 消息、22 路信号的实时双向传输。

- **通信协议**：设计 11 位优先级调度 CAN ID 分配方案（控制帧仲裁优先级高于数据帧），发布工业标准 DBC 配置文件，Python 脚本实现 JSON→C 头文件+DBC+信号描述符表一键生成，对接 Vector CANalyzer / SavvyCAN / cantools 工具链
- **实时固件**：FreeRTOS 7 任务 6 队列，按实时性分级调度（CAN 最高优先级防丢帧）。驱动层实现总线故障自动恢复（BusOff）、硬件 ID 掩码过滤、发送超时防护（5ms 重试+taskYIELD 防死锁）、TEC/REC 运行状态监控与周期诊断上报
- **分层中间件**：参考 AUTOSAR 规范自研 COM 层（信号描述符表驱动编解码，单工作缓冲区优化至 15 字节）+ RTE 层（真实 CAN/模拟数据热切换，开发调试零硬件依赖），实现端到端数据保护（有效性标志+4-bit 滚动计数器+3s 心跳超时存活检测，参考 E2E Profile 1）
- **UDS 诊断**：按 ISO 14229-1 标准实现 UDS 诊断栈，支持 6 种服务（会话控制/按 ID 读数据/读 DTC/清除 DTC/会话保持/ECU 复位），双节点独立物理寻址，含 S3 定时器与 tick 回绕安全防护
- **上位机应用**：Qt 5.15 仪表盘 UI，集成百度地图导航、4G 联网、WiFi 管理、倒车影像+超声波测距预警、车载音乐播放，CAN 测控数据经 COM 层 E2E 校验后实时驱动界面
- **工程化**：配置驱动代码生成、15 处编译期 static_assert 静态断言校验、ARM Linux/Yocto 交叉编译 + x86 本地开发双平台支持

## 面试口头介绍

> 60 秒自然口述版本，面试开场用。括号内为术语解释，可展开。

> 我做的是一个基于 CAN 总线的分布式智能座舱系统。下位机用两块 STM32 跑 FreeRTOS，分别模拟汽车的动力域和车身域 ECU——一个管发动机转速、油量、超声波倒车雷达，另一个管温湿度、空调风扇、车窗。中间跑 250kbps 的 CAN 总线，上位机是一块 IMX6ULL 工业 Linux 板子，跑 Qt 5.15 做的车载仪表盘。

> 这个项目我参考了 AUTOSAR 的架构思想，自己设计了一个四层通信栈——SWC 应用层、RTE 运行时环境、COM 信号层、Driver 驱动层。为什么这么做呢？如果不分层，上层 UI 代码直接操作 CAN 帧的字节偏移，换个传感器就得改所有页面代码。我通过一个信号描述符表把"车速=25km/h"这个信号自动映射到 CAN 帧的哪个字节、哪个位——上层只需要调一个 `sendSignal(SID_SPEED, 25)`，不关心底下怎么发的。

> 容错方面我做了三层：每个传感器帧带一个 valid 有效标志位——物理坏了就清零，接收端不显示僵尸数据；带一个 4-bit 滚动计数器——从 3 跳到 5 就知道第 4 帧丢了；心跳方面我不发独立心跳帧，任何一个上报帧到达就刷新定时器，3 秒收不到就触发离线告警。这三层合起来就是 AUTOSAR 里 E2E Profile 1 的核心思想——数据新鲜度加完整性加节点存活性。

> 协议方面我用 JSON 定义了所有 15 条 CAN 消息的布局，写了个 Python 脚本一键生成 C 头文件、信号描述符表、还有工业标准的 DBC 文件——DBC 可以直接加载到 Vector CANalyzer 或者 SavvyCAN 里做总线分析。这样改协议只改一个 JSON 文件，三方自动同步。

> 另外还实现了一个 UDS 诊断栈，就是修车师傅插诊断仪读故障码用的那个 ISO 14229-1 协议。六个服务——切换诊断会话、按 ID 读数据、读和清除故障码、ECU 远程复位。搭配着周期诊断帧实现了持续监控加按需诊断两套机制。


## 面试追问速答

### 基础层（项目概况、设计动机）

**Q: 这个项目一句话介绍？**
> 我用两块 STM32 模拟汽车 ECU，通过 CAN 总线把传感器数据发到 Linux 车机上，Qt 做仪表盘显示。自己设计了一套分层通信中间件，让上层代码不碰 CAN 帧的字节偏移。

**Q: 为什么选 CAN 总线，不是 SPI、I2C、以太网？**
> CAN 是汽车行业事实标准——多主架构、硬件仲裁、差分信号抗干扰、自带 CRC 和错误重传。SPI/I2C 是板级总线，距离超半米就不稳；以太网带宽高但实时性不如 CAN，而且车里已经有 CAN 了，加网关比换协议便宜。250kbps 对传感器数据绰绰有余——我的稳态负载才 2.5%。

**Q: 两个 STM32 节点怎么区分的？**
> 同一套代码通过 `#ifdef NODE_A` / `#ifdef NODE_B` 编译出两套固件。CAN ID 用 bit[10:8] 编码节点号——001 是 A（动力域）、010 是 B（车身域）。硬件滤波器也按节点配置——A 只收 0x080-0x081，B 只收 0x100-0x103。UDS 诊断地址也不一样——A 用 0x7E0/0x7E8，B 用 0x7E1/0x7E9。

**Q: Qt 端怎么跑起来的？开发环境是怎样的？**
> Windows 上写 STM32 固件（Keil MDK AC6 编译，CMSIS-DAP 烧录），Ubuntu 上写 Qt 终端（CMake+Ninja）。x86 开发时用 vcan0 虚拟 CAN，连 RTE 的 Mock 模式——不接硬件就能跑完整 UI 测试。部署到 ARM 车机用 Yocto SDK 交叉编译，SCP 传过去直接运行。

**Q: Mock 模式有什么用？**
> 开发调试不需要连硬件。RTE 切到 MOCK 模式后 50ms 定时器模拟档位(P→R→D 循环)、转速油量三角波、温湿度波动，而且有物理约束——P 档车速为 0、R 档限速 1/4，不是纯随机数。一键切换，改了 `setMode(MOCK)` 所有 UI 代码不动。

**Q: DBC 是什么？为什么重要？**
> CAN Database，汽车行业描述 CAN 消息的标准格式。定义每条消息的 ID、DLC、信号名、起始位、位宽、字节序、缩放因子、物理单位、值表枚举。我的 DBC 是 Python 从 JSON 自动生成的，可以加载到 Vector CANalyzer、SavvyCAN、cantools 里——全是工业标准工具。DBC 相当于 CAN 总线的"字典"——没有它只能看到一串十六进制。

**Q: 为什么不用 AUTOSAR 官方工具链？**
> AUTOSAR 商业工具像 Vector DaVinci 一套要几十万。我是理解了它的分层思想之后手工实现的——设计模式完全对标，只是代码手写而非工具生成。面试时说实话比吹牛更有说服力。

**Q: 企业级代码和原型代码有什么区别？**
> 原型代码假设一切正常——发送一定成功、传感器不会坏、对端不会死。企业级代码对每种已知故障模式都有显式的检测和处理。我的项目每条故障路径都有对应防护：发送失败有超时重试+yield、传感器坏了有 valid 标志清零、节点死了有心跳超时、CAN 帧丢了有 rolling counter 跳跃检测、收到畸形帧有 DLC 上下限截断、BusOff 有硬件自动恢复+错误回调。不是说写了多少行，而是覆盖了多少条故障路径。


### 进阶层（架构设计、通信协议）

**Q: 四层通信栈具体怎么分？**
> SWC 是 UI 页面，只收 Qt 信号不碰 CAN。RTE 做模式切换——开发时能一键切到 Mock 模式在 x86 PC 上跑完整 UI，不用连硬件。COM 层负责信号和 CAN 帧的双向映射，通过描述符表自动定位字节偏移。Driver 层封装 SocketCAN，对上是统一的 `sendFrame` 接口。分层的好处：换传感器只改描述符表一行，换通信介质只改 Driver，上层零改动。

**Q: 信号描述符表是什么？怎么工作的？**
> 就是一个 C 结构体数组，每条记录存了信号 ID、所属 CAN ID、在 8 字节 payload 里的起始字节、位宽、是否有符号、E2E 保护级别。发送时 `Com_SendSignal(SID, value)` 查表→写缓冲区→`Com_Flush()` 发出。接收时 `Com_ReceiveFrame` 存帧→`Com_ReceiveSignal(SID)` 反向解包。22 个信号共用这一张表，CAN ID 和字节偏移不出现在 SWC 代码里。

**Q: COM 层和 AUTOSAR COM 是什么关系？**
> 不是实现了 AUTOSAR，是参考了它的设计。AUTOSAR COM 的标准接口是 `Com_SendSignal(SignalId, data)` → 内部查 Signal Descriptor Table → 打包进 I-PDU → 触发发送。我的 COM 层接口和流程完全一致——`Com_SendSignal(SID, value)` → 查 `g_signal_desc[]` → 写 `g_tx_buf` → `Com_Flush()`。区别是 AUTOSAR 用商业工具从 DBC 生成代码，我是 Python 脚本从 JSON 生成。

**Q: 滚动计数器为什么用 4-bit 而不是 8-bit？**
> 我把 counter 和 valid flag 合在一个字节里——bit0 是 valid，bit[7:4] 是 counter，零额外带宽开销。4-bit 循环周期 16 帧，在 10Hz 发送频率下窗口 1.6 秒，足够检测连续丢帧。如果要检测更大的间隙——16 帧都丢了，心跳早就超时了。所以 counter 检测短期丢失，心跳检测节点整体掉线，两者互补。如果 counter 位宽再大，就需要多占 payload 字节——对一个 8 字节的 CAN 帧来说每一 bit 都是成本。

**Q: 心跳为什么不用独立帧，而是融合到数据帧里？**
> 最早设计用了独立心跳帧，1Hz 周期只发一个"我还活着"。问题有两个：占用了全总线最高优先级的 CAN ID，心跳帧丢了就误报。改成"任一上报帧到达就是心跳"——节点每 100ms 发 4 帧数据，任何一帧到达就 reset 计时器。好处：不依赖单一帧类型（4 种帧互为备份），不需要额外 CAN ID 和带宽，心跳频率等效 10Hz 远优于原来的 1Hz。

**Q: 这个项目的 CAN ID 分配是怎么设计的？**
> 11 位标准 ID 用 bit-field 编码：bit[10:8]=节点(001=A, 010=B, 011=诊断)，bit[7]=方向(0=控制, 1=上报)。为什么 bit[7]=0 给控制？CAN 仲裁是逐位线与——谁先发显性位(0)谁赢。所以控制帧 ID 从 0x080 起，上报帧从 0x180 起——控制指令的仲裁优先级永远高于传感器数据。倒车报警蜂鸣器是控制指令(0x080)，转速数据是上报(0x183)——报警不应该因为总线被转速刷屏而延迟。

**Q: 代码生成器是怎么工作的？**
> Python 脚本读 `can_signals.json`（15 条消息的完整信号定义），产出 4 个文件：`can_protocol_common.h`（CAN ID 宏 + frame 结构体）、`com_signal_defs.h`（信号 ID 枚举 + 描述符表 + COM API）、`uds_protocol.h`（UDS 常量）、`smart_cockpit.dbc`（标准 DBC）。Qt 端还有 15 个 `static_assert` 编译期校验 CAN ID 值是否和头文件一致。

**Q: 20KB SRAM 怎么省的？**
> 最早给每个 CAN ID 分配一个 8 字节 buffer——1024 × 8 = 8KB，占 SRAM 的 40%，编译直接报 L6406E。后来改成单工作缓冲区：一个 8 字节 `g_tx_buf` + CAN ID + DLC + 计数器，总共 15 字节。因为 100ms 周期内发完一帧再发下一帧，不需要同时缓存多帧。

**Q: ISR 和任务怎么通信？用了哪几种方式？为什么不同方式？**
> 三种。CAN 帧用队列——`xQueueSendFromISR` 入队，自动复制数据不共享指针。超声波用任务通知——`vTaskNotifyGiveFromISR` 比队列快 45%，还天然带超时机制（500ms 没收到回波就是传感器故障）。ADC 采样值用全局变量——单写者且非 ISR 上下文，不需要锁。选型逻辑：多写者必用队列，1 对 1 且要超时用任务通知，1 对 1 无超时的采样数据用全局变量（最省）。

**Q: CAN 滤波器怎么配的？**
> STM32F103 有 14 个滤波器组，我用了两个。FilterBank=0 用 32 位掩码模式——节点 A 只收 0x080-0x081（MaskId=0x7FE 忽略了 bit0），节点 B 只收 0x100-0x103（MaskId=0x7FC 忽略了 bit[1:0]）。不设滤波器所有总线 ID 都进 FIFO，6 帧深度瞬间满。FilterBank=1 精确匹配 UDS 诊断请求——节点 A 用 0x7E0，节点 B 用 0x7E1。

**Q: 你做这个项目遇到的最大难点？**
> CAN 发送死锁。CAN 任务最高优先级，BusOff 后邮箱不可用，while(1) 忙等不 taskYIELD——所有低优先级任务全部饿死，系统假死。后来加了 100 次重试上限 + 每 10 次 yield 一次。根因是 FreeRTOS 优先级的边界 case——不是低优先级卡高优先级，是最高优先级把自己卡死了，谁也救不了。


### 深入层（底层原理、协议细节）

**Q: CAN 总线仲裁到底怎么逐位竞争的？**
> CAN 用 CSMA/CR——载波监听、多主、逐位仲裁。每个节点发送的同时也在监听总线。两个节点同时发帧：ID 从高位到低位逐位比较，谁先发出隐性位(1)而对方发的是显性位(0)，谁就"输了"——主动退出发送，不破坏总线。赢的节点继续发完，输的节点等当前帧结束自动重试。整个过程不需要主节点协调，不丢数据。这也是为什么 CAN ID 越小优先级越高——ID 高位先比较，先遇 0 先赢。

**Q: CAN 位定时怎么算的？采样点为什么选 81.25%？**
> APB1 是 36MHz，预分频 9 得到 4MHz CAN 内核时钟。位时间 = SyncSeg(1TQ) + BS1(12TQ) + BS2(3TQ) = 16TQ = 4μs，波特率 250kbps。采样点在 13/16 = 81.25%，在 Bosch 推荐的 75-87.5% 范围内。CubeMX 默认配置采样点只有 62.5%，太靠前——总线振铃没稳定就读，长距离多节点容易出错。推到 81.25% 用 4 个额外 TQ 换更大噪声裕度。代价是 SJW 只有 1TQ，位同步窗口窄——一个稳健偏保守的选择。

**Q: DBC 的 Intel 和 Motorola 字节序有什么区别？**
> Intel（小端序 `@1+`）：从 start bit 开始，LSB 在低字节。比如 16-bit 信号 start=8，那 byte[1] 是低 8 位，byte[2] 是高 8 位。Motorola（大端序 `@1-`）：从 start bit 开始，MSB 在低字节。我全部统一用 Intel 格式——C 代码收发都是小端序（`buf[1] | (buf[2] << 8)`），DBC 也统一用 `@1+`，不会出现工具解码和 C 代码不一致的问题。

**Q: STM32 CAN 控制器的 TEC/REC 是什么？怎么用？**
> TEC（发送错误计数）和 REC（接收错误计数），STM32 的 bxCAN 在每个错误事件后自动增减。每检测到一个错误 +8，每成功一帧 -1。TEC>255 触发 BusOff（控制器完全断开总线）。我在 ErrorCallback 里读 ESR 寄存器提取这两个值，分了四个等级：<96 OK，≥96 Warning，≥128 Error Passive，BOFF 标志位 BusOff。每 10 秒通过诊断帧（0x300）上报。关键坑：TEC 是 8 位寄存器，最大值 255，不能判断 >255；必须检查硬件 BOFF 标志位。

**Q: UDS 诊断会话的 S3 超时怎么实现？防 tick 回绕吗？**
> 任何有效 UDS 请求到达时记录当前 `xTaskGetTickCount()`。主循环每 20ms 检查 `(当前 tick - 记录 tick) > 5000`。用差值比较——`uint32_t` 减法在回绕时自动正确：比如 tick 从 0xFFFFFFF0 跳到 0x00000100，0x100 - 0xFFFFFFF0 = 0x110，仍是正数。如果用绝对截止 tick（`记录 = tick + 5000`）来做 `当前 > 截止` 的比较，截止值本身会在加法时回绕，导致逻辑错误。

**Q: STM32 HAL 库有什么坑？这个项目遇到过什么？**
> 几个典型问题。一是 `HAL_CAN_AddTxMessage` 在 BusOff 时返回 `HAL_ERROR`，文档没写清楚需要多久恢复，直接 while(1) 重试会死锁整个系统。二是 CAN 滤波器的 `SlaveStartFilterBank` 在单 CAN 器件上被忽略但在双 CAN 器件上会影响滤波器分配——我直接去掉这个字段避免隐患。三是 `HAL_Delay` 和 FreeRTOS 的 SysTick 冲突——我的项目把 HAL 时基从 SysTick 改到 TIM4，SysTick 还给 FreeRTOS。四是 CubeMX 重新生成代码会覆盖 `USER CODE BEGIN/END` 之外的所有代码，稍不注意就丢修改。

**Q: DHT11 单总线读取为什么用 `vTaskSuspendAll` 而不是关中断？**
> DHT11 单总线对时序要求微秒级——起始信号拉低 18ms，响应脉冲 40μs 窗口。如果任务切换打断了脉冲读取，会误判 0/1。`vTaskSuspendAll()` 暂停调度但不关中断——HAL_Delay（基于 TIM4 中断）照常工作，只是不切换到其他任务。比关全局中断好，因为不影响 SysTick 和硬件中断的正常运行。

**Q: 如果这个项目节点数从 2 个扩展到 10 个，架构怎么调整？**
> 首先是 CAN ID 编码空间——当前 bit[6:0]=7 位只有 128 个信号 ID，10 个节点要扩大到更多位。其次是滤波器——STM32 只有 14 个滤波器组，10 个节点每个至少配一个，再加 UDS，可以合并掩码减少占用。然后是总线负载——10 个节点每 100ms 发 4 帧就是 400 帧/秒，在 250kbps 下负载约 12-15%，仍在安全范围内（<50%）。最后是心跳——3 秒超时要调宽松一些，因为帧更多仲裁延迟更大。架构上分层通信栈不用改——新节点配一个新描述符表项就行。

**Q: 如果继续做，你会怎么改进？**
> 四条线。第一，加应用层 CRC-8 实现完整的 E2E Profile 1——现在只有 counter+valid，CRC 可以防信号打包/解包时的字节错位。第二，升级 CAN FD——8 字节 payload 限制太死，诊断帧已经顶满，CAN FD 64 字节可以一个 UDS 响应传完 VIN。第三，双 CAN 总线+网关——实车动力域(500kbps)和车身域(125kbps)是分离的，中间网关做 ID 转换和速率适配。第四，UDS 安全诊断——加 0x27 SecurityAccess 做种子-密钥认证，刷固件和写配置需要先解锁。
